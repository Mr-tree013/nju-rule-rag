# 比赛定位调整：从"问答工具"到"学习产品"

> 日期：2026-05-26
> 背景：大一程序设计交互组比赛，目标获奖脱颖而出
> 剩余时间：1-2 周

---

## 一、现状诊断

### 当前项目的比赛劣势

- 本质是一个后端 API（`POST /ask`），缺少可交互的前端
- RAG 系统在比赛中过于常见，缺乏记忆点
- 评委看不到工程上的亮点（契约设计、验证脚本、评测管线）
- 交互组评委关注的是**交互体验**，不是代码质量

### 当前项目的优势

- 端到端 pipeline 已跑通（31 份校规 → 435 chunks → BM25+向量混合检索 → LLM 回答）
- 风险分级 + 拒答机制是差异化能力
- 知识库质量扎实，来源可追溯
- 工程基础好，加交互层的成本低

---

## 二、核心定位转换

| 之前 | 之后 |
|------|------|
| RAG 问答系统 | **校规互动学习平台** |
| "我问你答" | "我教你学 + 考考你" |
| 评委看代码逻辑 | 评委玩交互体验 |
| 定位：工具 | 定位：产品 |

**核心竞争力：** 把枯燥的校规变成新生可以"玩"的互动学习体验。别人搜东西，我们帮新生学会校规。

---

## 三、产品方案：双模切换

```
首页 ──→ 【问答模式】 ──→ 聊天式交互（复用现有 RAG 能力）
   │
   └──→ 【挑战模式】 ──→ 知识闯关（基于知识库自动出题）
```

两种模式共用一个知识库（435 chunks），共用一个 LLM pipeline，交互完全不同。

---

## 四、方案 A：极致交互包装（4-5 天）

### 目标
给现有 RAG 能力裹上一层**比赛级的交互外壳**，让评委能"玩起来"。

### 内容

| 模块 | 工时 | 说明 |
|------|------|------|
| Web 聊天界面 | 1-2 天 | 渐变色 + 毛玻璃风格，消息气泡，对标年轻化 AI 助手 |
| 风险可视化 | 0.5 天 | 每条回答顶部绿/黄/红三色指示灯，"高风险问题已自动警示"标签 |
| 来源追溯页 | 0.5 天 | 点击引用展开抽屉/弹窗，展示原文片段和来源文件 |
| 追问建议按钮 | 0.5 天 | 回答末尾自动生成 2-3 个追问按钮，一键触发下一问 |
| Demo 引导流 | 0.5 天 | 预置 6 个 Demo 问题，引导评委体验核心交互路径 |

### 三条 Demo 体验路径
1. **正常查询** → 看回答 + 追溯来源 + 追问按钮
2. **高风险拒答** → 红色警示 + 建议联系教务员，体现系统保守设计
3. **无依据拒答** → 灰色提示 + 追问建议，体现不编造的原则

---

## 五、方案 B：加上"挑战模式"（额外 2-3 天）

### 核心理念
从"我问你答"升级为"你考我"——用 RAG 的逆向应用作为差异化亮点。

### 功能设计

- **自动出题**：基于 435 个 chunk，LLM 自动生成选择题（单选/多选/判断）
- **答题交互**：用户选题后显示对错 + 校规依据原文引用
- **统计面板**：正确率、已掌握知识点、薄弱环节

### 出题策略
```
低风险话题（流程指引）：考"缓考在哪里申请？""补考截止时间？"
中风险话题（概念理解）：考"学业预警触发条件？""辅修学分要求？"
高风险话题（纪律处分）：不考（避免焦虑）
```

### 预期效果
- 比赛作品里突然有一个"能玩"的东西，评委眼前一亮
- 记忆点明确："那个让我做校规题的项目"
- 与学习定位高度吻合

---

## 六、决策建议

| | 方案 A 单独 | 方案 A + B |
|---|---|---|
| 工时 | 4-5 天 | 7-9 天 |
| 获奖把握 | 中上 | 高 |
| 风险 | 低 | 中（出题质量需调试） |
| 记忆点 | "做得漂亮" | "做得漂亮 + 能玩" |

**建议**：先做方案 A 保证基石，剩的时间冲方案 B。如果时间不够，方案 A 单独交付也足够竞争。
